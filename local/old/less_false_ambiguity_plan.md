# План: Снижение False-Ambiguous Rate в ambiguity check

## Диагностика проблемы

### Текущие результаты
- **68.18% false-ambiguous rate** (15 из 22 запросов помечены как ambiguous)
- Из 15 ложных срабатываний — все 15 запросов абсолютно однозначны и имеют единственный корректный SQL

### Анализ 15 ложных срабатываний

**Категория 1 — Конкретные числовые фильтры (очевидно OK):**
- Q13: топ-3 телефонов по отношению NumGE1500/NumTstTakr — метрика явно указана
- Q325: COUNT молекул с атомом 'n' и label '+' — конкретные фильтры
- Q1533: Consumption где Price/Amount > 29.00 и ProductID = 5 — всё указано
- Q1472: CustomerID с минимальным Consumption в LAM в 2012 — MIN явно подразумевается

**Категория 2 — Конкретные строковые значения (lookup по exact match):**
- Q498: перевод карты 'Tendo Ice Bridge' на французский — конкретная запись
- Q113: birth_date клиента по loan date '1996-01-03' и amount 98832 — конкретная запись
- Q685: ViewCount/DisplayName для поста 'Computer Game Datasets' — конкретная запись
- Q1365: expense_description для минимального remaining budget — однозначная агрегация
- Q1429: событие с VP на конкретном адресе — всё конкретно

**Категория 3 — Агрегации явно следуют из формулировки:**
- Q417: % expansion-сетов с японским переводом — ratio, явно задан
- Q1480: самый потребляющий месяц SME в 2013 — MAX(SUM) следует из "самый"
- Q1529: SUM(Price) и SUM для конкретного клиента — SUM явно указан
- Q1388: топ по SUM(amount) — агрегация явная
- Q1339: средняя стоимость для Elijah Allen в сентябре или октябре — AVG явно указан
- Q1359: отношение бюджетов двух конкретных событий — конкретные значения

### Корневые причины

**Причина 1: Промпт заточен под банковскую аналитику, BIRD — мультидоменный**
Все примеры в `AMBIGUITY_PROMPT_TEMPLATE` — про "отделения", "время ожидания", "точки".
LLM видит схемы molecules/cards/schools/gasstations и не может соотнести их с примерами
→ паникует → возвращает ambiguous.

**Причина 2: ambiguity check не получает знания о реальном содержимом БД**
`_check_ambiguity()` передаёт только `self.db_schema` (light: только имена/типы колонок).
НЕ передаёт:
- `self.relationships_str` (FK/PK — уже собирается в `build()`)
- `self.column_stats_str` (диапазоны, топ-категории — уже собирается в `build()`)

Без этого LLM не может проверить: "существует ли поле 'Charter Funding Type' с конкретными
значениями?" → неуверен → ambiguous.

**Причина 3: Порог принятия решения слишком низкий**
Текущая логика: "любое сомнение = ambiguous". Нужна логика: "ambiguous ТОЛЬКО если
существуют ≥2 принципиально разных SQL-интерпретации, дающих разные результаты".

**Причина 4: LLM не знает контекст бенчмарка**
BIRD-запросы специально сконструированы, чтобы иметь ОДИН правильный SQL. LLM об этом
не знает и применяет излишне строгий стандарт.

**Причина 5: Нет few-shot примеров из реальных BIRD-доменов**
Нет ни одного примера из химии, карточных игр, статистики школ — а именно из таких доменов
приходят запросы в BIRD.

## Шаги улучшений

### Шаг 1 (P0): Передавать в ambiguity check полные знания о БД

**Что:** Изменить `_check_ambiguity()` в `text2sql_implementation.py` — передавать в промпт
не только `db_schema`, но и `relationships_str` + `column_stats_str`.

**Почему это работает:** Зная, что `Charter Funding Type` — категориальный столбец с 3
уникальными значениями `('Directly funded', 'Locally funded', ...)`, LLM поймёт, что
фильтр по нему однозначен. Зная диапазоны числовых полей, поймёт, что числовые сравнения
конкретны.

**Изменения в `prompts.py`:**
Добавить в `AMBIGUITY_PROMPT_TEMPLATE` два новых placeholder'а:
```
### Связи между таблицами:
{db_relationships}

### Статистика столбцов (диапазоны, топ-значения категорий):
{column_statistics}
```

**Изменения в `text2sql_implementation.py`, метод `_check_ambiguity()`:**
```python
ambiguity_prompt = AMBIGUITY_PROMPT_TEMPLATE.format(
    db_schema=self.db_schema,
    db_relationships=self.relationships_str,   # добавить
    column_statistics=self.column_stats_str,   # добавить
    user_query=user_query,
)
```

**Ожидаемый эффект:** -25–35% false ambiguous rate

### Шаг 2 (P0): Переписать критерии принятия решения в промпте

**Текущая логика:** "Ответь ambiguous при любой неопределённости"

**Новая логика:**
> Запрос **однозначен**, если существует разумная SQL-интерпретация, которую написал бы
> квалифицированный аналитик данных, имея перед собой схему БД.
>
> Запрос **неоднозначен** ТОЛЬКО если существуют как минимум ДВЕ принципиально разные
> интерпретации (разные агрегации, разные таблицы, разный смысл результата), и нет способа
> выбрать одну без дополнительной информации от пользователя.

**Новые правила для промпта:**
1. Конкретные числовые значения, ID, строки, даты → **OK**
2. Агрегация явно следует из слов "топ", "самый", "минимальный", "максимальный", "средний",
   "сумма", "количество" → **OK**
3. Запрос можно выразить в SQL хотя бы одним разумным способом → **OK**
4. Если не уверен — **предпочитай OK**
5. **Ambiguous ТОЛЬКО если** оба условия:
   - Существуют ≥2 принципиально разных интерпретации
   - Эти интерпретации дают принципиально разные результаты

**Ожидаемый эффект:** -20–30% false ambiguous rate (в сочетании с Шагом 1)

### Шаг 3 (P1): Добавить разнообразные few-shot примеры

**Текущие примеры:** исключительно банковская аналитика ("отделения", "ожидание")

**Добавить примеры из BIRD-доменов:**
```
- Запрос: "How many schools have average math score above 560 and are directly funded?"
- Ответ: OK
(конкретный числовой порог + конкретное значение поля)

- Запрос: "What is the French translation for the card 'Tendo Ice Bridge'?"
- Ответ: OK
(конкретная запись + конкретный язык)

- Запрос: "Count distinct molecules with nitrogen atoms labeled as carcinogenic"
- Ответ: OK
(COUNT — однозначная агрегация, фильтры конкретны)

- Запрос: "Which customer in LAM segment had minimum consumption in 2012?"
- Ответ: OK
("minimum" → MIN агрегация, период указан, сегмент указан)

- Запрос: "Show me the best branches"
- Ответ: Неоднозначность: уточните метрику ("лучших" по какому показателю?) и период.
```

**Ожидаемый эффект:** -10–15% false ambiguous rate

### Шаг 4 (P1): Добавить контекст о природе запросов

Добавить в начало промпта блок:

```
### Контекст:
Запросы поступают из аналитического бенчмарка. Они сформулированы так, чтобы иметь
единственный правильный SQL-ответ. Если ты можешь написать разумный SQL без уточнений —
запрос ОДНОЗНАЧЕН. Не придумывай неоднозначности там, где её нет.
```

**Ожидаемый эффект:** -5–10% false ambiguous rate

### Шаг 5 (P2): "Оптимистичный fallback" — double-check механизм

Если после шагов 1–4 false-ambiguous rate всё ещё > 20%, добавить второй LLM-вызов:

**Идея:** Если ambiguity check вернул "ambiguous", попробовать сгенерировать SQL.
Если SQL успешно сгенерирован — значит запрос скорее всего однозначен.

```python
if ambiguity_check["ambiguous"]:
    # Попытаться сгенерировать SQL несмотря на флаг ambiguous
    sql_attempt = await self.generate_sql(user_query)
    if sql_attempt["status"] == "success":
        # SQL написан — запрос оказался однозначным
        return {"status": "success", "query": sql_attempt["sql_query"]}
    else:
        return {"status": "ambiguous"}
```

**Трейдоффы:**
- Удваивает стоимость обработки "ambiguous" запросов (доп. LLM-вызов)
- Может вернуть SQL там, где правда нужно было попросить уточнение (для Ambrosia-бенчмарка это минус)
- Для BIRD-бенчмарка — чистый выигрыш, поскольку там ambiguous = 0 баллов всегда

**Рекомендация:** Ввести переменную окружения `OPTIMISTIC_AMBIGUITY_FALLBACK=true/false`
и включать только для BIRD-mode.

**Ожидаемый эффект:** -10–20% false ambiguous rate

---

### Шаг 6 (P2): Добавить unique values низкокардинальных колонок в light schema

В `_get_db_schema_light()` добавить опцию для получения уникальных значений колонок с
distinct ≤ 15. Это те же данные, что уже собираются в `_explore_column_statistics()` для
категориальных колонок, только встроенные прямо в схему.

Альтернатива: убедиться, что `column_stats_str` достаточно полно описывает эти значения —
он уже включает `top` values для категориальных колонок с ≤10 уникальных значений.

## Итоговый вид нового `AMBIGUITY_PROMPT_TEMPLATE`

```python
AMBIGUITY_PROMPT_TEMPLATE = """
Ты — помощник, который проверяет запросы к базе данных на неоднозначность.

### Контекст:
Запросы поступают из аналитического бенчмарка и сформулированы так, чтобы иметь
единственный правильный SQL-ответ. Если можно написать разумный SQL без уточнений —
запрос ОДНОЗНАЧЕН. Предпочитай OK, если есть сомнения.

### Схема базы данных:
{db_schema}

### Связи между таблицами:
{db_relationships}

### Статистика столбцов (диапазоны значений, топ категорий):
{column_statistics}

### Правила принятия решения:

Отвечай "OK" если:
- Запрос содержит конкретные числовые значения, ID, строки, даты
- Агрегация следует из слов "топ", "самый", "минимальный", "максимальный", "средний",
  "сумма", "количество" и т.п.
- Запрос можно выполнить с разумным SQL хотя бы одним способом
- Ты не уверен — предпочитай OK

Отвечай "Неоднозначность: [краткое объяснение]" ТОЛЬКО если ОДНОВРЕМЕННО:
1. Существуют ≥2 принципиально разных интерпретации запроса
2. Эти интерпретации дают принципиально разные результаты
3. Нет способа выбрать одну без уточнения у пользователя

### Примеры:

Запрос: "Сколько школ с баллом математики > 560 напрямую финансируются?"
Ответ: OK

Запрос: "Какой перевод карты 'Tendo Ice Bridge' на французский?"
Ответ: OK

Запрос: "Какой клиент из сегмента LAM потратил меньше всего в 2012 году?"
Ответ: OK

Запрос: "Покажи топ-3 самых проблемных отделений по средней длительности ожидания в январе 2025"
Ответ: OK

Запрос: "Покажи список всех отделений"
Ответ: OK

Запрос: "Покажи худшие точки"
Ответ: Неоднозначность: уточните метрику ("худшие" по какому показателю?) и временной период.

Запрос: "Покажи точки по времени ожидания"
Ответ: Неоднозначность: уточните агрегацию (среднее / максимальное / минимальное)
и временной период.

### Запрос пользователя:
{user_query}
"""
```
